# Gear Spec And Detail Plan

## 背景

当前项目已经有装备库、装备墙、购买台账和分析看板，但装备层面的信息仍然偏基础：

- 装备库主要展示名称、品牌、型号、分类、价格和图片
- 装备墙主要强调档案状态与时间线
- 更细的选购参数还没有进入正式数据结构与浏览体验

这些参数平时不需要常驻在卡片中，但在挑选装备、回看装备、做购买判断时有明显价值。

项目当前已有适合承载这类信息的字段：

- `prisma/schema.prisma` 中 `GearItem.specJson`

因此后续更合理的方向不是继续往卡片里堆信息，而是：

1. 给不同装备类型增加结构化参数
2. 在装备库中提供图片点击后的快速详情层
3. 让参数成为“低频但高价值”的第二层信息

## 目标

- 不增加首页级卡片负担
- 不打乱现有“新增购买”主流程
- 提升装备库的判断信息密度
- 让装备图片不只是陈列，而能进入参数详情视角

## 建议范围

### 1. 装备参数体系

优先支持以下三类：

- 羽毛球
- 球拍
- 球鞋

其他分类先不做强 schema，保留通用扩展位。

### 2. 参数展示位置

不直接放入装备库卡片常驻显示。

优先放在：

- 装备库图片点击后的详情层
- 后续可能的装备详情页参数区

### 3. 参数维护入口

不在“新增购买”弹窗里维护。

原因：

- 新增购买的核心任务是补交易信息，不是编辑装备基础资料
- 装备参数应被视为库级数据，而不是单次购买数据

建议维护入口：

- 设置中的装备库维护
- 项目装备库数据源维护

## 参数建议

### 羽毛球

首批建议字段：

- 球速
- 毛片类型
- 球头类型
- 耐打定位
- 飞行稳定性
- 适用场景
- 建议温度/海拔说明

建议示例：

```json
{
  "speed": "77",
  "featherType": "goose",
  "corkType": "natural_cork",
  "durabilityLevel": "high",
  "flightProfile": "stable",
  "usageScene": ["training", "match"]
}
```

### 球拍

首批建议字段：

- 重量规格
- 平衡点
- 挥重
- 杆硬度
- 类型
- 中杆长度/拍框取向
- 建议磅数范围
- 双打/单打倾向

建议示例：

```json
{
  "weightClass": "3U",
  "balancePointMm": 302,
  "swingWeight": 88,
  "shaftFlex": "stiff",
  "playStyle": ["attack"],
  "frameProfile": "box",
  "tensionRangeLbs": "24-30"
}
```

### 球鞋

首批建议字段：

- 中底/发泡科技
- 是否碳板或稳定片
- 缓震取向
- 包裹取向
- 抓地取向
- 楦型
- 鞋面支撑科技
- 重量区间

建议示例：

```json
{
  "foamTech": ["boom"],
  "plateTech": ["carbon_plate"],
  "cushionProfile": "responsive",
  "fitProfile": "wrapped",
  "tractionProfile": "high_grip",
  "lastShape": "standard"
}
```

## 交互方案

### 装备库图片点击

建议做成“快速详情层”，而不是单纯图片预览。

交互：

1. 用户点击装备图片
2. 当前卡片图片做放大过渡
3. 弹出详情层
4. 左侧显示放大的装备图片
5. 右侧显示参数与补充信息

### 详情层内容结构

右侧建议顺序：

1. 名称 / 品牌 / 型号
2. 分类标签
3. 结构化参数
4. 参考价 / 最近购买 / 是否在心愿单
5. 主动作：`新增购买`

次动作可选：

- `加入心愿单`
- `移出心愿单`

### 动画建议

建议使用克制的共享元素放大，不要做过重转场。

重点：

- 图片从卡片内放大到左侧主视图
- 背景淡入
- 右侧参数分段淡入

目标是“更像看装备卡片展开”，不是“切到另一个页面”。

## 与现有逻辑的边界

### 不在新增购买里做的事

- 不编辑型号
- 不编辑品类
- 不编辑参数
- 不编辑封面图

新增购买只负责：

- 单价
- 数量
- 总价覆盖
- 购买时间
- 渠道
- 状态
- 备注

### 参数数据属于哪里

参数属于：

- 装备库数据
- `GearItem.specJson`

不属于：

- `PurchaseRecord`

## 技术落地建议

### 第一阶段

- 明确 `specJson` 的分类 schema
- 为羽毛球 / 球拍 / 球鞋建立字段映射
- 先支持只读展示

### 第二阶段

- 扩展项目装备库静态数据与设置维护入口
- 让 `/api/catalog` 返回 `specJson` 或整理后的详情字段

### 第三阶段

- 在装备库实现图片点击详情层
- 加入共享元素或近似放大动画

## 暂不执行的原因

- 当前更高优先级是把装备库、购买台账、装备墙的主流程继续收顺
- 参数体系一旦开始做，最好连同数据源、维护入口和详情层一起设计，不适合半做

## 后续实施顺序

1. 先补 schema 设计
2. 再补 catalog 数据源
3. 再做详情层 UI
4. 最后决定是否扩展到装备墙详情页
